# 仓库协作说明

`AGENTS.md` 是本仓库中供 Agent 执行的规范源。本文件是便于人工审阅的中文镜像，两者应保持一致；规则发生冲突时以 `AGENTS.md` 为准。

## Blog 工作流

- 涉及文章、草稿、分类、标签、资源、文章互链或全站写作约定时，使用 `.codex/skills/hesperine-blog-writer/SKILL.md`。
- 涉及论文阅读或以论文为中心的 Blog 时，同时使用 `.agents/skills/hesperine-paper-reading-blog/SKILL.md`。通用 skill 负责 Jekyll 和全站规范，论文 skill 负责资料核对、解释、证据、现状检查和工程理解。
- 用户对写法、理解、术语、文章范围和文章价值的更正，是改进该 skill 的重要依据。
- 当一次更正揭示了可复用的规则，应检查对应的 skill，并把整理后的规则写进去，让后续写作能够复用。
- skill 不是只能追加的反馈日志。应根据实际情况增加、改写、合并、收窄或删除规则；用户较新的明确更正可以取代旧规则。
- 不要把一次性的偏好或只适用于某篇材料的判断泛化成永久规则。只有底层原则可能反复适用时，才将其保留下来。
- 保持 skill 简洁且自洽。加入规则时，同时检查附近是否存在重复、冲突或已经应该删除的表述。

## 写作质量

- 按时间或技术谱系组织系列是一条有效线路，不能因为其中少数节点信息增量有限，就判断整条线路有问题。
- 文章的深度和篇幅应与节点的信息价值匹配。能提供独特思想、工程模式、关键实验或有证据支撑判断的内容应重点展开；信息增量较小但具有历史定位价值的节点应简写。
- 路线图是会持续调整的索引，不要求每一项占据同等篇幅。一个节点可以根据价值写成长篇精读、短文、其他文章中的背景段，或者暂缓处理。
- 优先依据一手资料、代码、实验和当前实践展开分析，并明确区分原论文当时的主张与今天仍然成立的部分。

## 仓库维护

- 保留工作区中与当前任务无关的修改；除非用户明确要求，否则不要暂存或提交。
- 除非用户明确要求公开，否则路线图和作者规划不要进入文章的可见正文。
- 修改后执行 repo-local blog writer skill 要求的校验。
